什么是 Agent？

一篇给普通人的扫盲贴，读完你也能说"就这？"

1 LLM：一台超级"接龙机"

大语言模型（LLM）——比如 ChatGPT、Claude、DeepSeek——最基础的本事就一个：你问，它答。

但它的运作方式跟你想象的可能不太一样。LLM 本质上是一台超级接龙机——你给它一段话，它预测下一个最可能出现的"词"是什么，接上去；然后把新句子再喂给自己，预测再下一个词……就这么一个字一个字往外蹦，直到它觉得"说完了"。

每次只预测下一个词，输出后又接回输入——循环往复，就连成了一整段话

这里说的"词"，在 AI 圈有专门的名字——token（词元）。一个 token 可能是完整词语，也可能是单字，甚至可能是词语的一小段（比如"苹果"可能拆成"苹"和"果"）。总之，它是 LLM 输入输出的最小单位。

那这台接龙机怎么造出来的？简单说：先拍脑袋设计一个极其巨大的数学方程——几百亿到上万亿个参数——然后用海量文字去"训练"它，不断微调参数，让接龙接得越来越准。人类已经没法手算这个过程了，全靠显卡硬堆。

它真有智能吗？——老实说，不知道。业界有很多解释（比如"涌现"），但没人能确切说清楚为什么接龙接到一定程度，模型就突然行为像"懂了"。这也解释了 幻觉——LLM 本质是预测下一个字，没有"真假"的概念，所以会一本正经地胡说八道。

2 聊天是怎么聊起来的

多轮对话：每次把历史全发一遍

LLM 本身没有记忆。每一轮对话在它看来都是独立的新问题。

那怎么聊？方法极为简单粗暴：每次提问，把之前的聊天记录原封不动全部重新发给它。上一轮它说过的、你说过的，打包在一起塞进去。这样 LLM 就"知道"在聊什么了。

代价显而易见——聊得越久，上下文越长，token 烧得越快。于是厂商搞了提示词缓存（prompt cache）：如果两次输入的前半截一样（历史记录部分通常不变），后半次可以蹭前半次的计算结果，这部分被复用的前缀就叫命中缓存（cache hit），命中的 token 可以享受大幅折扣。

人设怎么来的？

如果你直接问 LLM"你是谁"，它会老实回答自己的模型名。但各家聊天产品里的 AI 各有各的人设，靠的是在对话开头偷偷塞一段系统提示词（System Prompt）——你正常看不到它，但 LLM 会率先读到。相当于给 AI 的人设说明书。

那个"深度思考"是什么鬼？

你以为模型在深沉思考？其实还是接龙。只不过在正式回答之前，它先"自言自语"了一段推理过程——这段用特殊标记包着，不给你看。因为接龙时会参考前面所有内容，所以这段草稿会提升最终回答的质量。说白了：先打草稿，再交卷，假装草稿不存在。

3 让 AI 干活：工具调用

光聊天不行，AI 还应该能干实事——搜索网页、读写文件、发邮件。但 LLM 本质只会接龙，不能真去操作电脑。怎么办？找个中间人。

LLM 说暗号，中介执行，假装是 LLM 自己干的——用户全程无感

流程分四步：

  • ① 报菜名：中介通过系统提示词告诉 LLM——"我这有搜索、读文件、发邮件这些工具，要用就说暗号"。
  • ② 说暗号：LLM 接龙时觉得需要搜索，就输出特殊格式的"暗号"（比如一段特定结构的文字），表示"帮我搜 XXX"。
  • ③ 真执行：中介拦截到这个暗号，自己去调真正的搜索接口。
  • ④ 塞回去：中介把搜索结果写成一段话，塞回给 LLM。LLM 看到搜索结果，继续接龙回答你。

整个过程中，你就觉得 AI"自己去搜了一下"，浑然不觉背后有个中介在忙活。

4 Agent 循环：让工具调用转起来

上面的流程做了一次工具调用。但复杂任务可能要用十几次工具、分好几个阶段。

那把上面的流程循环起来就行了：中介不断把工具执行结果喂给 LLM → LLM 判断"还需要调工具"就继续发暗号 → "活干完了"就输出最终回答。这就是 Agent 循环（Agent Loop）。

核心就这么简单：LLM + 工具 + 循环 = Agent

给这个循环配上丰富多样的工具，再加一些上下文管理策略，你就得到了智能体（Agent）。到了这个阶段，LLM 主要的工作已经从聊天转向了规划 + 调用工具。

5 Agent 的关键配件

上下文压缩（Compact）

长任务执行到一半，上下文要爆了怎么办？让 LLM 自己写一份精简总结，把几十轮历史压缩成几段话。风险是 LLM 可能把关键信息也"精炼"没了。

自定义工具 & MCP

内置工具不可能面面俱到。你可以写自己的工具程序，然后通过 MCP（模型上下文协议） 这个标准接口对接给 Agent。MCP 就是一个"工具对接规范"，避免每家 Agent 各搞一套接口。

技能（Skill）

工具管"能操作什么"，技能管"怎么做更好"。技能是一份按需加载的说明书——Agent 先看"菜单"决定需要哪个技能，再加载对应内容的全文，避免一次性塞太多无关指令。

记忆（Memory）

每次任务结束，上下文就清空了。Agent 可以把重要的经验教训存下来，下次开工前先翻一翻"备忘录"。

子 Agent（Subagent）

复杂任务中有相对独立的子任务时，派一个新 Agent 去干，自己等汇报。本质就是外包——主 Agent 管规划，子 Agent 管执行。

6 Agent 之上，还有什么？

RAG：外挂知识库

严格来说，RAG 本身和 Agent 没什么关系——它是一种基于 LLM 的资料检索方法，多用来搭庞大的知识库系统。存资料时，系统把内容切片、算成"向量"（一串反映知识点的数字）存进数据库。提问时，同样算出问题的向量，去库里找最相似的几段，交给 LLM 总结回答。

但当 Agent 配上 RAG 就厉害了——干活之前，Agent 可以先调 RAG 系统"翻资料"，带着更多上下文再去执行任务。这就是 Agentic RAG。

Harness：给马套上缰绳

到这里，整套系统都是以 LLM 为绝对核心搭的——LLM 的接龙输出支配一切。但 LLM 的输出是有随机性的，天马行空、信马由缰。

怎么办？在 Agent 外面加一层非 LLM 的可靠规则，约束它的行为，确保它按照我们想要的方式工作。这套"缰绳"就叫 Harness——用代码逻辑而不是提示词来兜底。

OpenClaw：全自动助手

传统的 Agent 依然是人主导的——你打开它，给任务，它执行，然后结束。

但如果要当更聪明的个人助手，光被动接活不够。它需要能主动做事：

  • 传感器触发：监控某些事件，事情发生就自动执行对应任务。
  • 定时执行：每到特定时间，自动跑固定流程。
  • 自我进化：不断总结以往经验，记成记忆，越来越了解你。

从"人触发任务"变成"自动触发任务"，再加上记忆环节持续自我迭代——这就是 OpenClaw 的思路。

工作流：组团干活

有些任务大到单个 Agent 搞不定了。那就给多个 Agent 安排分工，不同 Agent 各司其职，按流程接力完成——这叫 Agent 编排（Orchestration）。

设立执行流程，让不同 Agent 按照顺序执行、交接成果，这就是 工作流（Workflow）。说白了就是：分工 + 制定阶段和流程。到这个阶段，你关心的不是单个 Agent 怎么实现，而是如何组合一群 Agent 形成一套解决问题的机制。

Loop 工程：让工作流也"活"起来

既然 Agent 层面可以引入主动执行和自我进化，那工作流层面自然也可以。让整个工作流不再只是被动完成任务，而是能主动触发、自我迭代和优化——这就是 Loop 工程。

∞ 总结

Agent 不是什么魔法。剥开来看就一句话：

LLM 接龙 + 中介 + 工具 + 循环 = Agent

往上再加记忆、技能、子 Agent、RAG、多 Agent 编排……万变不离其宗。

祛魅完毕。下次有人跟你吹 Agent 多玄乎，你可以微微一笑："不就一台接龙机套了个循环嘛。"
